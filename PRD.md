# PRD: open-pos-analytics

**作成者**: Ryusei | **日付**: 2026-07-09 | **ステータス**: ドラフト
**アイデアID**: #2998（主軸）+ #3008 + #3018 の統合 | **ドメイン**: open-pos Extension/Analytics | **難易度**: Medium〜High

> next-top20.md の統合推奨に基づき、3つの分析系アイデアを1リポジトリのモジュール群として実装する:
> - **heatmap**（#2998 POS-Heatmap、canonical 30点・Tier S）: 時間帯×曜日の売上ヒートマップ
> - **basket**（#3008 POS-Basket-Insight、27点）: バスケット（併売）分析
> - **cohort**（#3018 POS-Customer-Cohort、25点）: 顧客コホート/RFM分析
>
> ※ phase2（競合検証）/phase3（アーキテクチャ検証）の既存データはこの3件には存在しないため、本PRDで新規に記述した。

## 問題定義

open-pos（v2.0.0）は販売・在庫のトランザクション処理に集中しており、蓄積された売上データを**意思決定に変換する層が無い**。小規模店舗の店主は「いつ忙しいか」「何と何が一緒に売れるか」「客が離脱していないか」を経験と勘で判断しており、大手小売が持つ分析基盤（棚割り・販促・CRM分析）との格差が大きい。

open-pos-analytics は open-pos の売上データを読み取り専用で取り込み、**店主が5分で読める形**の3つの分析ビューを提供する。

## 目標と成功指標

| 目標 | 指標 | ターゲット |
|------|------|----------|
| open-pos データからの分析パイプライン確立 | 取込済みトランザクションでの3モジュール動作 | サンプルデータ10万件で全ビュー描画 < 3秒 |
| 繁忙パターンの可視化（heatmap） | 時間帯×曜日ヒートマップの精度 | 集計値がSQL直叩きと100%一致 |
| 併売発見の実用性（basket） | 併売ペア抽出（支持度・信頼度・リフト値） | リフト値上位20ペアの妥当性を手検証 |
| 顧客理解（cohort） | 月次コホートのリピート率・RFMセグメント | 会員IDあるデータでセグメント3分類が機能 |
| ポートフォリオ品質 | make check / make quality 全パス、テスト付き | CI green、CLAUDE.md ≤ 50行 |

## スコープ外（MVP）

- リアルタイムストリーミング分析（バッチ/オンデマンド集計のみ）
- open-pos への書き戻し（発注提案・価格変更等のアクション実行）
- 多店舗横断の比較分析（単一店舗スコープ）
- POPテンプレート自動生成（#3008の第2段階機能。分析結果のCSV/PNGエクスポートまで）
- 祝日・イベント特異日検出のML化（MVPは祝日カレンダー静的突合）

## ユーザーストーリー

1. **店主**として、時間帯×曜日の売上ヒートマップを見たい。なぜなら**シフトを繁忙時間に合わせて組みたい**から
2. **店主**として、カテゴリ別のピーク時間帯を知りたい。なぜなら**惣菜の仕込み時間と品出しを最適化したい**から
3. **店主**として、併売率の高い商品ペアを知りたい。なぜなら**関連陳列で客単価を上げたい**から
4. **店主**として、休眠しかけている常連客のセグメントを見たい。なぜなら**離脱前に販促を打ちたい**から
5. **開発者（自分）**として、open-pos のデータを読み取り専用で安全に取り込みたい。なぜなら**本体の運用に影響を与えたくない**から

## 技術要件

### 技術スタック

- **Backend**: Kotlin / **Quarkus 3.34**（分析API、open-pos データ取込）
  - ※ アイデア原文と ideas-tech-reeval は Spring Boot 案だが、open-pos 本体・エコシステム（pos-voice-concierge 等）が Quarkus で統一されているため Quarkus を採用する。**ADR-0001 として記録すること**
- **Frontend**: TypeScript / React (Vite) + D3.js（ヒートマップ・コホートチャート・併売ネットワーク図）
  - ※ 原文は Next.js 案。ダッシュボードSPAでありSSR不要のため、テンプレート標準の Vite 構成を採用。ADRに含める
- **DB**: PostgreSQL（取込トランザクション・集計マテリアライズドビュー）
- **連携**: open-pos の PostgreSQL リードレプリカ想定の読み取り接続、または売上エクスポートAPI/CSV取込（Issue #2 で接続方式を決定しADR化）
- **Infra**: GCP Cloud Run（コンテナ、Ship段階で Dockerfile 追加）

### アーキテクチャ

```
open-pos (PostgreSQL) --读取--> [ingest] --正規化--> analytics DB (PostgreSQL)
                                                        |
                        backend (Quarkus): /api/heatmap /api/basket /api/cohort
                                                        |
                        frontend (React+D3): Heatmap / Basket / Cohort ビュー
```

- 分析は集計テーブル（時間帯×曜日×カテゴリ、ペア共起、顧客×月次）を事前計算し、APIは読み取りのみ
- モジュール間はスキーマを共有（transactions, line_items, customers）し、集計ロジックのみ独立

### 非機能要件

- パフォーマンス: 10万トランザクションで各ビューAPI p95 < 500ms（事前集計前提）
- セキュリティ: open-pos 接続は読み取り専用ロール。会員IDは分析DB内でハッシュ化（PII非保持）。カード情報は一切扱わない（PCI-DSSスコープ外を維持）
- スケーラビリティ: 単一店舗・3年分（~100万行）までをMVP設計範囲とする

## 競合分析

※ phase2 データ無しのため新規調査が必要（Issue #2 に含める）。既知の競合状況:

| 競合 | 概要 | ギャップ |
|------|-----|---------|
| Airレジ / スマレジの分析機能 | クラウドPOS付属の売上分析 | 自社POS専用・ロックイン。open-pos エコシステムには存在しない |
| Square Analytics | POSと一体の分析ダッシュボード | 同上。バスケット分析・RFMは上位プランのみ |
| BIツール（Metabase等）自前構築 | 汎用BI | 店主がSQL/モデリングを書く必要がある |

**差別化ポイント**: open-pos ネイティブ（スキーマ理解済み・ゼロ設定取込）+ 小規模店特化のプリセット分析（設定不要で3ビューが出る）。OSS。

## 技術リスク

| リスク | 発生可能性 | 影響 | 軽減策 |
|--------|----------|------|--------|
| open-pos スキーマへの密結合（本体変更で壊れる） | 中 | 中 | 取込層でDTOに正規化し、本体スキーマ依存を ingest モジュールに閉じ込める |
| 会員ID無しデータではコホート分析が成立しない | 高 | 中 | cohort モジュールは会員データ有無を検出し、無い場合は「匿名リピート推定不可」を明示するデグレード動作 |
| バスケット分析の組合せ爆発（SKU数大） | 低〜中 | 中 | 支持度の最小閾値でプルーニング（Apriori系の標準手法）。MVPはペア（2-item）のみ |
| D3.js 実装コスト | 中 | 低 | ヒートマップ→コホート→ネットワーク図の順で難度が上がるため、この順に実装 |
