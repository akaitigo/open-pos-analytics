# ADR-0003: 会員IDは SHA-256 + 環境変数ソルトでハッシュ化し生値を保持しない

- **日付**: 2026-07-09
- **ステータス**: 承認済み
- **決定者**: Ryusei

## コンテキスト

cohort/RFM 分析には顧客の同一性が必要だが、分析DBに PII（生の会員ID）を持つとリスクとコンプライアンス負担が増える。分析要件は「同一顧客の識別」のみで「誰か」を知る必要はない。

## 決定

- 取込時に `SHA-256(salt + ":" + member_id)` の16進64文字を `customer_hash` として保存する
- ソルトは環境変数 `ANALYTICS_CUSTOMER_SALT` から取得。**prod プロファイルではデフォルト無し**（未設定なら起動失敗）。dev/test のみ `dev-only-salt` を既定値とする
- 生の会員ID・氏名・連絡先・カード情報はどのテーブルにも保存しない（スキーマにカラム自体を設けない）。テストで「生IDがDBに存在しない」ことを検証する

## 結果

- 逆引き（ハッシュ→会員）は open-pos 側で同一ソルトを使って再計算しない限り不可能。販促アクション（#11 Phase 1）で会員へ届ける必要が生じた場合は、open-pos 側でハッシュ照合する設計とし、分析DBには生IDを持ち込まない
- ソルトローテーションは顧客同一性の断絶を意味するため原則行わない（実施時は全量再取込）

## 却下した選択肢

- 生ID保存 + アクセス制御: 漏えい時の影響が大きく、MVP の分析要件に不要
- HMAC-SHA256: saltの用途としては同等。鍵管理をシンプルに保つため単純連結+SHA-256 を採用（衝突・伸長攻撃が成立する文脈ではない）
