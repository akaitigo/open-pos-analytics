# ADR-0002: open-pos データ接続は CSV エクスポート取込を MVP 方式とする

- **日付**: 2026-07-09
- **ステータス**: 承認済み
- **決定者**: Ryusei

## コンテキスト

open-pos の売上データを分析DBへ取り込む方式として3案を比較した。

| 案 | 利点 | 欠点 |
|----|------|------|
| A. PostgreSQL リードレプリカ直結 | リアルタイム性、実装少 | open-pos 内部スキーマへ密結合（本体変更で即破損）。レプリカ運用が単一店舗MVPには過剰 |
| B. open-pos エクスポートAPI | 疎結合、認可を本体に委譲 | open-pos 側に未実装のAPIが必要（本体への変更が前提） |
| C. CSV エクスポート取込 | 本体無変更で今日動く。スキーマ結合が「CSV列定義」1点に固定される。テスト容易 | バッチ（リアルタイム不可）。手動/cron 運用 |

## 決定

**MVP は C（CSVエクスポート取込）**。PRD の非機能要件（バッチ集計前提、リアルタイム不要）に合致し、open-pos 本体への変更ゼロで進められる。

- CSV 列定義（1行 = 1明細）: `transaction_id,occurred_at,member_id,product_code,product_name,category,quantity,unit_price`
- `occurred_at` は ISO-8601 オフセット付き（例 `2026-07-01T18:05:00+09:00`）
- `member_id` は空欄可（非会員）。取込時に SHA-256 ハッシュ化し生値は保存しない（ADR-0003）
- 取込は冪等: `source_transaction_id` 既存のトランザクションはスキップ
- MVP の CSV はクォート無し前提（商品名にカンマを含めない）。open-pos 実データで必要になれば RFC4180 対応に拡張

## 結果

- スキーマ依存は `ingest` パッケージ（パーサ+DTO）に閉じ込める。将来 B 案（API）へ移行してもパーサ差し替えのみ
- Phase 1 で open-pos 側にエクスポートAPIを追加する場合、本 ADR を改訂する
